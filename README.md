# distribute / install

A pair of Perl scripts for securely distributing and installing files and
packages across multiple hosts. distribute.pl packages and signs files on
a central server and pushes them to remote hosts via rsync; install.pl
verifies signatures and installs them, integrating with syslock for
immutability management.

Primary platform is OpenBSD. install.pl also runs on Linux and macOS for
installing architecture-independent packages (Perl scripts and modules).
distribute.pl runs on OpenBSD only (uses OpenBSD ports packages as source).

Also included in this repo:
- **gendoas.pl** — generates per-host doas.conf files from a shared template
  (included in the distribute package)
- **pkg_info.pl** — minimal pkg_info equivalent for Linux and macOS
  (included in the install package)

The OpenBSD packages are signed with signify. To verify:
```
signify -C -p discord.org-2026-pkg.pub -x install-1.0.tgz
signify -C -p discord.org-2026-pkg.pub -x distribute-1.0.tgz
```
Public key: https://www.discord.org/lippard/software/discord.org-2026-pkg.pub

## Overview

The distribute/install system provides a complete chain of custody for files
deployed across a fleet:

1. distribute.pl packages plain files, packages, or custom-generated content
   into signed tarballs on the central server
2. Files are pushed to `/var/install/<hostname>/` on each target host via
   rsync using the [rsync-tools](https://github.com/lippard661/rsync-tools)
   `_rsyncu` infrastructure
3. install.pl runs on each target host, verifies signatures, unlocks
   immutability flags as needed, installs, and re-locks

All distributed content is signed with signify. The recommended key naming
convention is `<domain>-<year>-pkg` (e.g., `example.com-2026-pkg.sec`).
install.pl verifies signatures before installing anything, and applies
different trust levels depending on whether the signing key matches the
expected domain key.

## Tools

### distribute.pl

Reads `/etc/distribute.conf` to determine what to distribute and to which
hosts. For each named item in the config, packages the content into a
signed gzipped tar archive and rsyncs it to `/var/install/<hostname>/` on
each target host.

**File types**:
- `plain` — a file distributed as-is (config files, scripts, certificates,
  individual programs)
- `package` — a pre-built signed OpenBSD package `.tgz` from the local
  ports packages directory; verified before distribution, installed by
  install.pl using pkg_add (OpenBSD) or the built-in minimal pkg_add
  (Linux/macOS)
- `custom` — content generated by built-in subroutines; currently
  implemented: `doas.conf` (generated per-host from a template via
  gendoas.pl) and `ip-address` (updates config files when the home
  network WAN IP address changes)

**Signing key management**: distribute.pl defaults to an annual signify
key named `<domain>-<year>-pkg` derived from the hostname's domain, but
this can be overridden via `default-signing-key:` in the config or the
`-k` flag on the command line. If the selected key does not follow the
`<domain>-<year>-pkg` naming convention, distribute.pl warns and prompts
before continuing, noting that plain/custom files signed with a
non-standard key will require explicit user acceptance on the install side.
Warns in December if next year's public key is not yet present. Aborts if
the current year's signing key is missing.

**Command line**:
```
distribute.pl [-4|-6|-p|-k keyname|-c config|-h hosts|-d] item [item ...]
```

Options:
```
-4          Use IPv4 for rsync (default)
-6          Use IPv6 for rsync (for ip-address changes when IPv4 is down)
-p          Sign with prior year's domain key (mutually exclusive with -k)
-k keyname  Sign with specified key (filename or filename.sec; key must
            have its private key in /etc/signify). Overrides the default
            signing key. Mutually exclusive with -p.
-c config   Use specified config file (default: /etc/distribute.conf)
-h hosts    Distribute only to specified hosts (comma-separated)
-d          Debug output
```

Arguments are one or more named items from the config file.

### install.pl

Runs on a target host. Looks for files in `/var/install/` (or
`/var/installation/` on macOS), verifies signatures, and installs.
Integrates with syslock to unlock immutability flags before installation
and re-lock afterward. Appends a record of what was installed to
`/etc/CHANGELOG` (configurable to your own changelog location).

**Package types handled**:
- `<hostname>-<date>-<time>-package.tgz` — plain/custom package created
  by distribute.pl; verified and extracted to filesystem root
- `<name>-<version>.tgz` — OpenBSD package; verified then installed via
  pkg_add (OpenBSD) or built-in minimal pkg_add (Linux/macOS)

**Securelevel check**: on OpenBSD, install.pl checks kern.securelevel
before attempting to unlock immutability flags. It will not run (unless
`-f` is used) if securelevel > 0, since schg flags cannot be unlocked.
Intended to run in single-user mode or on systems using only uchg flags.

**Command line**:
```
install.pl [-f|-n|-d]
```

Options:
```
-f    Force: proceed even if securelevel > 0, assuming groups use uchg.
      Uses syslock audit mode to verify groups are actually unlocked.
-n    No syslock: skip all immutability flag management
-d    Debug output
```

install.pl takes no file arguments; it processes everything in the install
directory.

### gendoas.pl

Generates per-host doas.conf files from a shared template. The template
uses comment-based directives to mark which sections apply to which hosts:

```
# hosts: all
# permit nopass _rsyncu as root cmd /usr/local/bin/rsync args --server ...

# hosts: host1 host2
# permit nopass admin

# hosts: none
# This section is documentation only and will not appear in any generated config.
```

`hosts:` values can be `all`, `none`, a space-separated list of hostnames,
or `all except` followed by a list. Lines beginning with `# permit` or
`# deny` are uncommented when included; continuation lines (ending with
`\`) are handled correctly.

gendoas.pl reads from `/etc/doas.conf.template` by default and writes to
`doas.conf` in the current directory (use `-o` to specify output path).
On the local host it also installs directly to `/etc/doas.conf` with a
backup. Specifying a hostname argument generates a config for that host
instead of the local host.

Uses pledge/unveil on OpenBSD.

```
gendoas.pl [-o output_file] [hostname]
```

distribute.pl invokes gendoas.pl automatically for each target host when
distributing the `doas.conf` custom type.

---

### pkg_info.pl

A minimal pkg_info equivalent for Linux and macOS, working with the
`/var/db/pkg` package registration database created by install.pl's
built-in minimal pkg_add. Lists installed packages or shows details for
a specific package, using the same output format as OpenBSD's pkg_info.

```
pkg_info.pl              # list all installed packages (brief)
pkg_info.pl <package>    # show description for package
```

## Configuration: distribute.conf

Default location: `/etc/distribute.conf`

**Global section** (before any `name:` entries):

```
# Macro definitions (usable in any field value as %%MACRONAME%%)
GENDOAS="/usr/local/bin/gendoas.pl"

# All hosts distribute.pl knows about
host-list: host1, host2, host3

# All syslock groups that may be referenced in syslock-groups: fields
syslock-groups-list: etc, local, reportnew, rsync, signify, ssl

# Directory containing OpenBSD ports packages (amd64 default)
pkg-dir: /usr/ports/packages/amd64/all

# Default signing key (optional; overrides domain-derived default).
# $DOMAINNAME and $YEAR are available as special variables in this field only.
# default-signing-key: $DOMAINNAME-$YEAR-pkg
```

**Per-item entries** (one block per named item):

```
name: <item-name>        # used on distribute.pl command line
file: <source-path>      # source file or package name (<name>-PKG for packages)
dest: <dest-path>        # optional; destination path if different from source
type: plain|package|custom
syslock-groups: <groups> # optional; comma-separated syslock groups to unlock
hosts: all|host1,host2|all except host3
```

For `custom` type only:
```
unveil: <path>[,<path>]          # paths to unveil read-only
unveil-exec: <path>[,<path>]     # paths to unveil for execution
custom-vars: key=value[,key=value]  # variables passed to custom handler
```

**Special variables** usable in `file:` and `dest:` fields:
- `$SIGNIFY_PUB_KEY` — current year's signify public key path
- `$SIGNIFY_PUB_KEY_NEXT` — next year's signify public key path
- `$HOST` — target hostname (substituted per-host at distribution time;
  useful in `file:` for ip-address custom type to pull per-host source
  files from backup directories)

Macros defined with `MACRONAME = "value"` can be referenced as
`%%MACRONAME%%` in any field value.

**Package name convention**: packages are specified as `<name>-PKG` in
the `file:` field; distribute.pl finds the latest version of that package
in the configured packages directory.

## Signify Key Conventions

The recommended key naming convention is `<domain>-<year>-pkg`, where
domain is derived from the hostname's last two components. For a host
named `foo.example.com`, the default keys are `example.com-2026-pkg.sec`
and `example.com-2026-pkg.pub`.

The default signing key can be overridden in `distribute.conf` using
`default-signing-key:`, which accepts `$DOMAINNAME` and `$YEAR` as
special variables. The `-k` flag on the command line overrides this
further for a single run; `-k` and `-p` are mutually exclusive.

If the selected signing key does not follow the `<domain>-<year>-pkg`
naming convention, distribute.pl warns and prompts:

```
Warning: Signing key 'mykey' doesn't follow the standard pattern.
Expected pattern: example.com-YEAR-pkg (e.g., example.com-2026-pkg)

Plain/custom files signed with this key will require explicit user
acceptance during installation on target systems.

Continue with non-standard signing key? [y/n]:
```

**distribute.pl** accepts packages from any source for distribution,
provided the signing key's public key is present in `/etc/signify` and
follows the `-pkg` naming convention for the current or prior year.

**install.pl** applies different trust levels depending on content type:

For **packages** (OpenBSD `.tgz` files installed via pkg_add or the
built-in minimal pkg_add): any valid `-pkg` key in `/etc/signify` for
the current or prior year is accepted without prompting. Package signing
is an independent process; the distribution key acts as a transport
signature and packages may legitimately originate from multiple sources.

For **plain and custom files** (content generated and signed directly by
distribute.pl): files signed with the host's own domain key are installed
without prompting. Files signed with a different key — even one present
in `/etc/signify` — trigger an explicit confirmation prompt:

```
======================================================================
SIGNATURE VERIFICATION WARNING
======================================================================
File is signed with: otherdomain.com-2026-pkg.sec
Expected signing key: example.com-2026-pkg.sec

The signing key is valid and in /etc/signify,
but it is not the expected key for this system.

Accept this key and install the file? [y/n]:
```

The prompt requires an explicit `yes`, `no`, `y`, or `n` followed by
Enter; no default is accepted. This asymmetry reflects that plain and
custom files are directly authored content where a foreign signing key
is genuinely suspicious, unlike packages which may come from any
trusted source.

**Key security**: the signing key passphrase should not be stored on the
distribute server itself. The recommended approach is to store it in a
password manager (e.g., KeePassXC) protected by both a master passphrase
and a hardware key (e.g., YubiKey), unlocked only when signing.

## Minimal pkg_add / pkg_delete (Linux and macOS)

install.pl includes a built-in implementation of pkg_add and pkg_delete
for Linux and macOS, enabling installation of architecture-independent
packages (Perl scripts, modules, and similar tools) on non-OpenBSD systems.

The minimal pkg_add:
- Requires `@arch *` in the package `+CONTENTS` (architecture-independent)
- Installs files under `/usr/local`
- Remaps Perl module paths from OpenBSD's
  `libdata/perl5/site_perl` to the appropriate location on Linux or macOS
- Installs platform-specific sample configs: if the package contains a
  file named `macos.<configname>` or `linux.<configname>`, that file is
  used as the default config instead of the OpenBSD one
- Creates `/var/db/pkg/<package>/` registration entries (+CONTENTS, +DESC,
  +DISPLAY) compatible with OpenBSD's pkg database format
- On upgrade, runs minimal pkg_delete on the old version first, preserving
  any sample config files that have been modified from the installed version
  (checked by size and SHA256)
- Respects `@mode` settings in `+CONTENTS` for file and directory permissions
- Will not install the same version again or downgrade in place

pkg_info.pl reads the same `/var/db/pkg` registration database.

## rsync Infrastructure

distribute.pl pushes files to target hosts using rsync. By convention,
target hostnames have `-distribute` appended for the rsync destination
(e.g., `host1-distribute`). These should be defined as SSH config aliases
in `~/.ssh/config` on the distribute server, pointing to the `_rsyncu`
user with the appropriate SSH key and with rrsync enforced via
`authorized_keys` on the remote end.

Example `~/.ssh/config` entry:
```
Host host1-distribute
    HostName host1.example.com
    User _rsyncu
    IdentityFile ~/.ssh/id_ed25519_distribute
```

The `-6` flag appends `-distributev6` instead, for use when only IPv6
connectivity is available (e.g., during a WAN IP address change when the
new IPv4 address has not yet propagated).

See [rsync-tools](https://github.com/lippard661/rsync-tools) for the
rrsync and rsync-client/server infrastructure used on the remote end.

## syslock Integration

distribute.pl sends a signed `.grp` file alongside plain and custom
packages listing the syslock groups that need to be unlocked for
installation. install.pl reads and verifies this file, then calls
`sysunlock -g <group>` for each group before installing and
`syslock -g <group>` after.

For packages, install.pl assumes the `etc` and `local` syslock groups
(covering `/etc` and `/usr/local`).

The `-f` flag allows install.pl to proceed when securelevel > 0,
assuming the relevant groups use uchg flags (which can be unlocked
without single-user mode). It uses `sysunlock -a -q -g <group>` to
audit that groups are actually unlocked after the sysunlock call.

See [syslock](https://github.com/lippard661/syslock) for group
configuration and flag management.

## Installation

### Recommended: OpenBSD signed packages

Two packages are provided:

**install** (install.pl + pkg_info.pl):
```
pkg_add ./install-1.0.tgz                    # OpenBSD
```
Or using install.pl on OpenBSD, Linux, or macOS once it is itself installed.

**distribute** (distribute.pl + gendoas.pl):
```
pkg_add ./distribute-1.0.tgz                 # OpenBSD
```
The default distribute.conf included with the package references the
installed path for gendoas.pl automatically.

install.pl is the bootstrap tool — install it first, then use it to
install distribute and all other packages on OpenBSD, Linux, and macOS.

### Manual installation

```sh
# distribute.pl — on the central distribution server (OpenBSD)
cp src/distribute.pl /usr/local/sbin/distribute.pl
chmod 755 /usr/local/sbin/distribute.pl
cp etc/distribute.conf /etc/distribute.conf
chmod 600 /etc/distribute.conf

# install.pl — on each managed host
cp src/install.pl /usr/local/sbin/install.pl
chmod 755 /usr/local/sbin/install.pl

# gendoas.pl — on the distribution server
cp src/gendoas.pl /usr/local/bin/gendoas.pl
chmod 755 /usr/local/bin/gendoas.pl
cp etc/doas.conf.template /etc/doas.conf.template
chmod 600 /etc/doas.conf.template

# pkg_info.pl — on Linux or macOS hosts
cp src/pkg_info.pl /usr/local/bin/pkg_info.pl
chmod 755 /usr/local/bin/pkg_info.pl

# Create install directory on each managed host
mkdir -p /var/install        # OpenBSD/Linux
mkdir -p /var/installation   # macOS
```

## Security Notes

- `/etc/distribute.conf` should be mode 0600 (root only); it reveals
  your host topology, sync relationships, and file locations
- distribute.pl and install.pl are mode 0755; both require and verify
  that they are run as root
- The signing key secret file should be 0600 and its passphrase stored
  securely off the distribution server (see Key Security above)
- install.pl verifies signatures twice for OpenBSD packages installed
  via the built-in minimal pkg_add: once before reading the tar file and
  once after, to mitigate TOCTOU race conditions
- Sample config files that have been modified since installation are
  preserved during package upgrades; modification is detected by both
  size and SHA256 comparison
- The `/var/install` directory on target hosts should be accessible only
  to the `_rsyncu` user for writing (via rrsync restriction) and to root
  for reading
